  Summary of fixes

  1. Eshell up-arrow history (root cause found and fixed)

  Root cause: eshell-on-input-line? in qt/commands-core.ss used
  (qt-plain-text-edit-cursor-position ed) which returns a byte offset from

  character offset. The eshell welcome text "gsh ‚ÄĒ Gerbil Shell" contains an em
  dash (‚ÄĒ = 3 UTF-8 bytes), causing byte position > character length. This threw
   an exception caught silently by execute-command!'s with-catch, so the eshell
  branch was skipped and normal cursor-up ran instead.

  Fix: Rewrote eshell-on-input-line? to use Scintilla's line-based APIs
  (SCI_LINEFROMPOSITION + SCI_GETLINECOUNT) which avoid the byte/char mismatch
  entirely. The eshell-current-input and eshell-replace-input! functions were
  already safe (they operate on the full Scheme string from
  qt-plain-text-edit-text, not on byte positions).

  2. PNG crash (root cause found and fixed)

  Root cause: cmd-find-file and cmd-find-file-at-point in qt/commands.ss never
  checked image-file?. PNG files were opened as regular text:
  read-file-as-string read the binary data as a string, which was fed to
  Scintilla, causing a crash.

  cmd-find-file-at-point. Image files are now routed to qt-open-image-inline!

   buffers.
